# 05 — Memory Schema (Neo4j + ChromaDB)

The memory store has two coordinated layers: **Neo4j** for structured, stable relationships (people, medications, routines, places) and **ChromaDB** for unstructured **episodic** memories (things that happened). A blended query pattern combines them: vectors surface semantically relevant episodes; the graph ties them to entities, edges, and timelines.

---

## Graph layer — Neo4j (running locally)

### Node labels

| Label | Key properties | Notes |
| --- | --- | --- |
| `Person` | `id`, `name`, `relationship` (daughter, doctor, neighbor…), `notes` | The people in the user's life. Names are core relational data — **kept**, not stripped (see PII rules below) |
| `Medication` | `id`, `name`, `dose`, `schedule` (e.g. `["08:00","20:00"]`), `purpose` | Purpose kept vague if sensitive (see PII) |
| `Routine` | `id`, `label` (morning walk, tea time), `time`, `recurrence` | Anchors "what usually happens when" |
| `Place` | `id`, `label` (home, Dr. Rao's clinic), `type` | For "where" questions |
| `Object` | `id`, `label` (blue umbrella, house keys), `usualLocation` | Populated by perception ("where are my keys?") |
| `User` | `id`, `displayName`, `preferences` | The primary user profile |
| `Episode` | `id`, `text`, `timestamp`, `importance`, `source`, `piiFiltered` | References to ChromaDB for the vector embedding |

### Relationship types

| Relationship | From → To | Properties | Meaning |
| --- | --- | --- | --- |
| `KNOWS` | `:User` → `:Person` | `closeness` (string) | Relationship |
| `TAKES` | `:User` → `:Medication` | `schedule` (string[]) | With time schedule |
| `HAS_ROUTINE` | `:User` → `:Routine` | — | Daily routine |
| `LOCATED_AT` | `:Object` → `:Place` | `lastSeen` (datetime) | Last known location |
| `CARES_FOR` | `:Person` → `:User` | — | Caregiver link |
| `RELATED_TO` | `:Person` → `:Person` | `relation` (string) | e.g. Emily is Dr. Rao's patient |
| `INVOLVES` | `:Episode` → (`:Person`/`:Medication`/`:Object`/`:Place`) | — | Links an episode to entities |

### Cypher query examples

**Find medication status:**
```cypher
MATCH (u:User {id: $userId})-[t:TAKES]->(m:Medication {name: $medName})
RETURN m, t.schedule
```

**Find where the user's keys are:**
```cypher
MATCH (o:Object {label: "keys"})-[l:LOCATED_AT]->(p:Place)
RETURN p.label AS place, l.lastSeen AS lastSeen
ORDER BY l.lastSeen DESC LIMIT 1
```

**Find upcoming routines:**
```cypher
MATCH (u:User {id: $userId})-[r:HAS_ROUTINE]->(rt:Routine)
WHERE rt.time >= $now AND rt.time <= $endOfWindow
RETURN rt ORDER BY rt.time
```

## Vector layer — ChromaDB (running locally)

ChromaDB stores episodic memories as vector embeddings. Each episode is one document with metadata.

### Collection: `episodic`

```
Collection: episodic
  - embedding: float[] (generated by gemini-embedding-2 via API)
  - documents: text summary of the episode
  - metadata:
      episodeId: string      // matches Episode.id in Neo4j
      namespace: string      // for scoped retrieval
      timestamp: number      // epoch ms
      importance: number     // 0.0–1.0 scored by Gemma 4 at write
      entityIds: string[]    // Neo4j node IDs this episode involves
      source: string         // local_perception | cloud_perception | voice_note | caregiver_seed
      piiFiltered: boolean
```

### Vector namespaces

Namespaces in ChromaDB act as partitions, so retrieval can scope by kind:

| Namespace | Contents | Typical query |
| --- | --- | --- |
| `episodic/daily` | Day-to-day events | "What did I do today?" |
| `episodic/people` | Interactions tied to people | "When did I last see Emily?" |
| `episodic/objects` | Object sightings/locations | "Where did I put my umbrella?" |
| `episodic/medical` | Medication-taking events | "Did I take my pills?" |

### ChromaDB query example (JavaScript)

```typescript
const collection = await client.getCollection({ name: 'episodic' });
const results = await collection.query({
  queryEmbeddings: [embedding],
  nResults: 5,
  where: {
    $and: [
      { namespace: { $eq: 'episodic/medical' } },
      { timestamp: { $gte: todayStart } },
    ],
  },
});
```

## Blended query pattern (GraphRAG equivalent)

For a query like *"Did I take my morning pills?"*:

1. **Graph-first** (fast, exact): query Neo4j — `MATCH (u:User)-[:TAKES]->(m:Medication) WHERE m.schedule CONTAINS 'morning' RETURN m`
2. **Vector** (scoped to `episodic/medical`, today's window): query ChromaDB — semantically matching "took pills / medication" for today.
3. **Blend**: if a matching `:Episode` exists with a timestamp in the morning window → "Yes, at 08:15." If not → "I don't have a record of it — want to mark it now?"

For *"Where are my keys?"*: `:Object{label: "keys"}-[:LOCATED_AT]->:Place` from Neo4j, corroborated by the most recent `episodic/objects` episode mentioning keys from ChromaDB.

The blending logic lives in the `queryMemory` tool implementation (see [06-agent-loop.md](06-agent-loop.md)).

## PII filtering rules

The tension: names like "Emily" / "Dr. Rao" are exactly what the graph needs, but are also personal. **Strip only very sensitive info**, keep core relational data.

**Kept (core relational, needed for usefulness):**

- First names / relationship labels of people in the user's circle.
- Medication names and schedules (the app's whole job).
- Place labels, routines, object labels.

**Stripped / never stored (very sensitive):**

- Government / national IDs, insurance numbers, account numbers.
- Full street addresses and precise geolocation (store coarse label like "home" / "clinic" instead).
- Financial figures, card numbers.
- Free-form medical diagnoses/record numbers beyond the medication name+purpose needed to function.

The PII filter runs **on-device (Gemma 4 via Ollama)** at write time, before any Neo4j write, ChromaDB write, or cloud call. Every stored record carries `piiFiltered: true`.
